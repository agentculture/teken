# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Status

Early alpha. Published as `teken` on PyPI (once Trusted Publisher is configured). Users install with `uv tool install teken`; the command is `teken`.

> **Renamed from `afi-cli`.** The canonical distribution is now `teken` (Hebrew תֶּקֶן, "standard"). `afi-cli` remains a thin wrapper distribution (`packaging/afi-cli/`) that depends on `teken==<same version>`, kept in lockstep by the `version-bump` skill and a CI guard, so `uv tool install afi-cli` still works. The `afi` command is a deprecated alias for `teken`. The import package and command moved `afi` → `teken`; the dot-directory is `.teken/` (existing `.afi/` trees are still detected). A GitHub repo rename is a separate, later step — see the wrapper README and the PyPI Trusted Publisher notes.

Stack (mirrors `../culture` conventions for the AgentCulture OSS org):

- **Python 3.12+**, `uv` for deps, `hatchling` build backend.
- CLI: `argparse` with noun-based subcommand groups (`teken <noun> <verb>`), registered in `teken/cli/`.
- Tests: `pytest` (+ `pytest-xdist`, `pytest-cov`) under `tests/`.
- Lint: `flake8` + `flake8-bandit` + `flake8-bugbear`, `pylint`, `bandit`, `black`, `isort` (all in the `dev` dependency group).
- CI: `tests.yml` (PR), `publish.yml` (TestPyPI on PR, PyPI on push to main — Trusted Publishing / OIDC), `security-checks.yml` (scheduled weekly).
- **Version bump required on every PR**, even docs-only or trivial changes — enforced by the `version-check` job in `tests.yml`. PyPI history and `CHANGELOG.md` track each merged PR 1:1. Use the repo-local `version-bump` skill (`.claude/skills/version-bump/`) or run `python3 .claude/skills/version-bump/scripts/bump.py {major,minor,patch}` directly; pipe a JSON changelog object via stdin so `CHANGELOG.md` gets a populated entry. Default to `patch` for docs/config/CI; `minor` for new commands or features; `major` for breaking changes.

## Context

This repo belongs to the **AgentCulture** OSS org. Before making design decisions, read [`docs/agentculture.md`](./docs/agentculture.md) (what the org is, that agents are first-class members, and teken's foundational role) and [`docs/agent-first.md`](./docs/agent-first.md) (the paradigm — learnability on CLI, minimalism on MCP, discoverability on HTTP). Every feature proposal is expected to pass an Agent First review.

## Intended Purpose

**teken** = *Agent First Interface CLI*. A scaffolding tool that generates **all three interface surfaces** an AI agent may need to consume a tool, from a single source of truth:

1. **CLI** — with a built-in `learn` affordance so an agent can introspect the tool and author its own skill/usage notes (rather than relying on `--help` text alone).
2. **MCP server** — exposing a deliberately **minimal menu** of tools (surface area tuned for agent consumption, not maximal API coverage).
3. **HTTP site** — serving markdown pages plus a sitemap, so any agent with a fetch tool can navigate the docs without a bespoke client.

The "Agent First" framing is not just protocol breadth — it is **opinionated best-practice guidance baked into each generated interface** (learnability on CLI, minimalism on MCP, discoverability on HTTP). When proposing structure, optimize for agent ergonomics (discoverability, introspection, low menu cardinality) over human-UI conventions.

teken is expected to **dogfood itself**: its own CLI/MCP/HTTP surfaces should be generated by teken once bootstrapped. Design internal abstractions with that self-hosting round-trip in mind.

## Common commands

| Task | Command |
|---|---|
| Install deps | `uv sync` |
| Run CLI from source | `uv run teken ...` or `uv run python -m teken ...` |
| Run tests (parallel) | `uv run pytest -n auto -v` |
| Run a single test | `uv run pytest tests/test_cli.py::test_learn_subcommand -v` |
| Coverage | `uv run pytest --cov=teken --cov-report=term` |
| Lint Python | `uv run flake8 --config=.flake8 teken tests` |
| Format | `uv run black teken tests && uv run isort teken tests` |
| Lint markdown (auto-fix) | `scripts/lint-md.sh` |
| Install pre-commit hooks | `uv run pre-commit install` |
| Build distribution | `uv build` |

End users install with `uv tool install teken` — that's the supported install path, not `pip install`. (`uv tool install afi-cli` still works via the compatibility wrapper.) Building distributions: `uv build` produces `teken`; `uv build packaging/afi-cli -o dist/` produces the `afi-cli` wrapper. CI builds and publishes both.

## Markdown Linting

- **Auto-fix (recommended):** `scripts/lint-md.sh` — runs `markdownlint-cli2 --fix` across all tracked `.md` files (or pass paths to scope). The `lint-markdown` skill under `.claude/skills/` wraps this one-liner.
- **Pre-commit hook:** `.pre-commit-config.yaml` runs `markdownlint-cli2` in check-only mode. Activate with `pre-commit install` after installing [pre-commit](https://pre-commit.com/).
- **Config:** `.markdownlint-cli2.yaml` at the repo root (MD013/MD060 disabled). Lives in-repo because `markdownlint-cli2` stops walking at the git root and will not pick up any user-home config from inside the repo.

## Permissions

If you add project-local Claude Code tool permissions, put them in `.claude/settings.local.json` (per-user, gitignored) rather than modifying global settings. Prefer narrow patterns (e.g. `Bash(python3 scripts/lint-md.sh)`) over wildcards (`Bash(python3 *)`) — least privilege.
